# ActionsPlane — Security & Performance Review

Review date: 2026-05-24. Scope: `src/actionsplane/`, `migrations/`. **Review only — no source modified.**

This tool holds a GitHub App private key and can (in a later phase) open PRs org-wide, so the bar is high. The good news: the genuinely dangerous primitives (webhook HMAC, JWT minting, private-key handling, SQLAlchemy usage, YAML parsing) are implemented correctly. The real risks are (1) a completely open API surface (not-yet-built authn — a gap, not a bug), (2) a couple of audit-logic false negatives that *under-report security risk*, and (3) serial, non-rate-limited, unbounded GitHub I/O that will not scale past a small org.

## 1. Summary table (sorted by severity)

| Severity | Area | File:line | Issue | Fix |
|---|---|---|---|---|
| **High** | security (gap) | `api/app.py:41-181` | No authentication/authorization on *any* API endpoint. `/api/v1/findings`, `/audit/scorecard`, `/templates`, `/drift` expose the full org supply-chain posture (every unpinned action, broad-permission workflow) to anyone who can reach the port. This is a known not-yet-built gap, but it must gate any non-localhost deploy. | Add auth middleware (session/OIDC or signed token) before exposing beyond loopback. At minimum document "bind to localhost only" until built. No CORS config present either — add explicit allowlist when a browser origin is introduced. |
| **High** | security | `audit/pins.py:85-87` | Pin-classifier **false negative**: any ref that is not a SHA, not a `vN` tag, and not in the hardcoded `_BRANCH_NAMES` set is classified `TAG_PINNED`. A mutable branch like `@release` / `@stable` / `@my-feature` is therefore reported as MEDIUM, not the HIGH that a branch pin warrants. This understates real supply-chain risk (the exact tag/branch-retargeting class the module exists to catch). | Don't assume "unknown ref == tag." Treat anything that is not SHA-pinned and not a strict semver tag as at least branch-equivalent (HIGH), or resolve the ref type via the GitHub API. The 5-name branch allowlist is far too narrow. |
| **High** | perf/scale | `sync/worker.py:61-114` | `reconcile` and `audit_all` iterate **all** watched repos **serially** over one httpx client, with no concurrency, no pagination, no conditional requests, and no rate-limit handling. For audit, each repo does `list_workflow_files` + N× `get_file_text` sequentially. At ~3-5 calls/repo this blows the 5000/hr installation REST budget around a few hundred repos and the cron simply stalls. | Bound concurrency with `asyncio.Semaphore` + `asyncio.gather`; honor `X-RateLimit-Remaining`/`Retry-After`; add ETag/`If-None-Match` conditional requests (304s don't count against the limit); paginate `list_workflow_runs`; consider the GraphQL batch path the docstrings promise. |
| **Med** | security | `github/client.py:66-71` | `get_file_text` interpolates `path` straight into the URL (`/contents/{path}`) with no encoding or `..` rejection. Today `path` originates from GitHub's own listing so it's low-risk, but a `subpath` containing `..`/`%`/spaces would mis-target the contents API, and the pattern invites SSRF/traversal if `path` ever becomes user-supplied (e.g. a future template-binding `path`). `owner`/`repo` are likewise uninterpolated-but-unvalidated. | `urllib.parse.quote(path, safe="/")`; reject `..` segments and absolute paths; validate `owner`/`repo` against `^[A-Za-z0-9._-]+$`. |
| **Med** | security | `audit/engine.py:51-56` | Broad-permissions check is incomplete: `is_broad` only flags `write-all` and `contents: write`. It misses other dangerous grants — `id-token: write` (OIDC cloud-cred theft), `packages: write`, `actions: write`, `pull-requests: write`, `deployments: write`. These are exactly the scopes that turn a compromised action into lateral movement. False negative on real risk. | Flag any `*: write` that isn't on a small safe set, and treat `id-token: write` specially (HIGH). |
| **Med** | security | `audit/engine.py:58-67` | Missing-`permissions` finding is suppressed if **any** job declares permissions (`job_has_perms = any(...)`). A workflow with one scoped job and one unscoped job (which then inherits the write-all default on older repos) is silently passed. | Evaluate per-job: flag each job that has no effective `permissions` and no workflow-level default. |
| **Med** | perf | `db/repository.py:101-105` + `api/app.py:123-135` | `open_findings` returns an **unbounded** result set (no limit), and `/findings` filters `severity`/`finding_type` **in Python** after fetching every open finding for the repo/org. On a large org this loads thousands of rows per request. | Push `severity`/`finding_type`/limit into the SQL `WHERE`/`LIMIT`. Add a composite index `(repo_id, resolved_at)` and `(resolved_at, severity)` on `audit_findings`. |
| **Med** | perf | `events/bus.py:33-43` | `publish` opens (and closes) a **new Redis connection per event** when no connection is injected — which is how `worker.process_event` calls it (line 44/48). Under run/job webhook volume this is connection churn on the hot path. | Reuse a process-wide `aioredis` client (lazy singleton like `sync/queue.py` does for arq), or pass `ctx`-held connection through. |
| **Med** | perf/correctness | `api/app.py:170-178` | SSE `event_generator` has no client-disconnect detection. If a browser drops, the `subscribe()` async generator's `finally` (unsubscribe/aclose) only runs when the generator is GC'd/closed — with many reconnecting dashboards this leaks Redis psubscriptions/connections. | Wrap with disconnect handling (check `request.is_disconnected()` or rely on `EventSourceResponse`'s cancellation and ensure the `async for` is cancel-safe); cap concurrent subscribers. |
| **Med** | perf | `db/repository.py:125-138` | `resolve_stale_findings` passes the entire `seen_fingerprints` set inline via `notin_(...)`. For a repo with hundreds of findings this is a large `NOT IN (...)` literal every audit. The `or [""]` guard also means "no findings seen" resolves everything correctly but via an awkward sentinel. | For large sets, resolve via "last_seen_at < this_run_started" instead of `NOT IN`; stamp a run timestamp and resolve `last_seen_at < run_ts AND resolved_at IS NULL`. |
| **Med** | security | `ingestor/app.py:45` | `await request.body()` reads the full webhook body with no size cap before signature verification — an unauthenticated caller can stream a huge body to exhaust memory (DoS). Signature check happens after the full read. | Enforce a max content-length (GitHub payloads are <25MB; reject larger) at the ASGI/proxy layer and in-app. |
| **Low** | security | `ingestor/app.py:60` | `json.loads(body)` is not guarded — a signed-but-malformed body (only possible if secret leaks) yields an unhandled 500 rather than 400. Minor; signature gate protects it. | `try/except json.JSONDecodeError -> 400`. |
| **Low** | correctness | `github/app_auth.py:32-33` | `build_app_jwt` backdates `iat` by 60s, but when a caller passes `now` it backdates *again*; also `exp = iat + 600` means effective lifetime is ~9 min not 10. Harmless (within GitHub's 10-min cap) but slightly surprising. | Compute `exp` from un-backdated now, or document the intent. |
| **Low** | perf | `db/models.py:38` / `migrations/0001` | `repos.watched` is filtered on every reconcile/audit (`list_repos(watched_only=True)`) but has no index. Small tables today; add a partial index `WHERE watched` if repo count grows. | `CREATE INDEX ... ON repos (watched) WHERE watched`. |
| **Low** | perf | `github/client.py:39-49` | `list_workflow_runs` fetches a single page of `per_page` (default 50) with no pagination loop — reconcile can silently miss older dropped runs beyond page 1. Acceptable for a "recent" safety net but worth noting. | Document the single-page intent, or paginate with a bounded cap + `created` cursor. |

## 2. Top 5 fixes to do first

1. **Lock down the API** (`api/app.py`). It is fully unauthenticated and serves the org's complete security posture. Bind to localhost-only and/or add auth before any shared deploy. (High)
2. **Fix the pin classifier's "unknown ref = tag" assumption** (`audit/pins.py:85-87`). Mutable branches are being downgraded to MEDIUM — the tool under-reports the exact attack it targets. (High)
3. **Make reconcile/audit_all concurrent + rate-limit-aware** (`sync/worker.py`). Serial unbounded GitHub I/O will exhaust the 5000/hr budget and stall the cron on a real org. Add `gather` + semaphore + ETags/304s + pagination. (High)
4. **Broaden the permission audit** (`audit/engine.py:51-67`) to cover `id-token: write` and other `*: write` scopes, and evaluate missing-permissions per-job. (Med, security false negatives)
5. **Encode/validate the contents-API path and owner/repo** (`github/client.py`). Cheap hardening that closes a latent SSRF/traversal vector before `path` becomes user-supplied. (Med)

## 3. Things checked and found correct (already solid)

- **Webhook HMAC** (`ingestor/signature.py`): SHA-256 over the raw body, `hmac.compare_digest` (constant-time), rejects missing/badly-prefixed headers, and `ingestor/app.py:47-52` **fails closed** with 503 when no secret is configured and 401 on mismatch. Applied to every event before any processing. Correct.
- **GitHub App JWT** (`github/app_auth.py`): RS256, short TTL, `iat` backdated for clock skew, `iss` = app id. Standard and correct.
- **Token handling**: installation tokens cached with a 60s expiry skew (`InstallationToken.is_expired`), private key read from a *path* (never inlined), `github_webhook_secret` declared `repr=False`. No tokens, JWTs, secrets, or webhook payloads are logged anywhere (the one log line, `audit/service.py:36`, logs only owner/repo/path). Verified via grep across `src/`.
- **SQL safety**: all queries use the SQLAlchemy expression language / `pg_insert`; no raw SQL, no f-string-built queries, parameterized throughout. Upserts use Postgres `ON CONFLICT` correctly for idempotent at-least-once delivery.
- **YAML parsing** (`audit/parser.py`): `YAML(typ="safe")` — no arbitrary object construction. The YAML-1.1 `on:`→`True` quirk is handled. Non-mapping roots raise. Correct and safe.
- **DB session lifecycle** (`db/base.py`): async engine/sessionmaker are lazy + cached; `get_session` is a proper `async with` dependency. Engine is asyncpg (no blocking driver inside async). No obvious blocking calls inside async paths.
- **Audit service resilience** (`audit/service.py:35`): a single unparseable workflow is caught and skipped rather than aborting the whole-repo audit.
- **Finding fingerprinting** (`audit/findings.py`): stable sha256 dedup key, unique-constrained in the DB (`migrations/0002`), with sensible reopen-on-reappear and resolve-when-absent lifecycle.
- **Core indexes** present: `repos.installation_id`, `workflow_runs.(repo_id, workflow_id, created_at)`, `workflow_jobs.run_id`, `audit_findings.repo_id`, `template_bindings.(repo_id, template_id)`. The hot list/order paths are covered (gaps noted above are incremental).
- **Executor** (`executor/campaigns.py`): genuinely unimplemented (`NotImplementedError`), so the org-wide PR/write path is not yet a live risk; `bulk_edits_enabled` defaults `False`. Re-review when Phase 4 lands.
