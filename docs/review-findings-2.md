# ActionsPlane — Security & Performance Review #2

Review date: 2026-05-25. Scope: `src/actionsplane/`, `migrations/`. **Review only — no source modified.**

This is a follow-up to `docs/review-findings.md` (2026-05-24). The codebase has grown: API bearer auth (`api/auth.py`), the campaign/PR executor (`executor/*`), GitHub write methods (`github/client.py`), a GitLab provider (`providers/gitlab/`), and the drift sweep. Most of the prior review's quick-wins landed. The serious live risks are now in the **executor write path** and the **token-cache staleness** introduced by the multi-repo campaign loops.

Important correction to the review brief: there is **no `bounded_gather` / per-task-session concurrency refactor** in `sync/worker.py`. The worker is still fully serial (single `httpx` client, sequential `for repo in ...` loops in `reconcile`, `audit_all`, `drift_sweep`). So "did the concurrency refactor introduce races / pool exhaustion?" — there is no refactor to have introduced them. The prior High finding about serial, non-rate-limited GitHub I/O is therefore **still open** and unchanged.

## 2. NEW findings (sorted by severity)

| Severity | Area | File:line | Issue | Fix |
|---|---|---|---|---|
| **High** | perf/correctness | `executor/campaigns.py:23,51` + `github/factory.py:31-37` | **Stale-token bug in campaign loops.** `run_dry_run`/`apply_campaign` build a `cache: dict[int,str]` of installation tokens and `client_for_installation` stores the *raw token string* and **never checks expiry** (it only checks `is None`). Installation tokens live ~1h; a campaign that walks hundreds of repos serially over the (uncapped, non-rate-limited) GitHub API can easily exceed an hour, after which every subsequent repo gets a 401 and is marked `failed`/`conflict`. The `InstallationToken.expires_at` info is discarded at factory.py:34. | Cache the `InstallationToken` object (token + `expires_at`) and re-mint when `is_expired()`. Same fix applies to `worker._client_for` (worker.py:97-102) and `reconcile`'s `token_by_install`. |
| **High** | security | `executor/campaigns.py:46-49` + `api/app.py:228-241` | **Apply is gated only by a process-wide boolean, not by who/what triggered it.** `apply_campaign` checks `get_settings().bulk_edits_enabled` (a cached env flag) and nothing else. The `/api/v1/campaigns/{id}/apply` endpoint is auth-gated *only if `ACTIONSPLANE_API_TOKEN` is set* — which is **off by default**. So in the default config, anyone who can reach the port can `POST .../apply` and, if `bulk_edits_enabled=true`, open org-wide PRs with **no human-approval step, no per-campaign authorization, no record of `created_by`** (the column exists at models.py:139 but is never populated — repository.py:219-225 omits it). "Human-triggered" is asserted in the docstring but not enforced. | Require API auth to be configured before `bulk_edits_enabled` can take effect (fail-closed); record and require an approver (`created_by` + an explicit `approved_by`/approve step distinct from create); consider a second confirmation token on apply. |
| **High** | security | `executor/service.py:79-81` + `github/client.py:112-119` | **Re-apply / partial-failure leaves a poisoned branch and is not idempotent.** `open_pr_for_edits` always does `create_branch(... branch=f"actionsplane/{operation_id}")` where `operation_id = f"{op}-{campaign.id}"` — **constant per campaign**. `create_branch` POSTs `git/refs` which **422s if the ref already exists**. So any re-apply (retry after a transient failure, or a second apply click) throws on `create_branch`, the target is marked `conflict`, and if a previous partial run already pushed commits to that branch the new edits are silently never applied. There is also no cleanup of half-written branches. | Make branch names unique per attempt (include a short hash/timestamp) or detect-and-reuse an existing branch (GET ref, update if present); treat "ref exists" as resumable, not a conflict; consider deleting the branch on full failure. |
| **Med** | security | `executor/campaigns.py:64` | **apply recomputes from HEAD and silently re-pins to whatever HEAD now resolves to.** The comment "recompute fresh" is correct in spirit (avoids stale blob SHAs), but the resolver in `_resolve_pin_refs` (service.py:28-45) re-runs `get_commit_sha(owner,repo,ref)` against the **action's** repo at apply time. If an upstream tag was retargeted between dry-run and apply, the PR pins to the *new* (possibly malicious) SHA without surfacing that the SHA differs from the previewed diff. The dry-run preview the human approved is not what necessarily gets committed. | Pin to the exact SHAs captured in the dry-run (store the resolved map on the target), or diff the recomputed edits against the previewed diff and re-require approval if they differ. |
| **Med** | security | `executor/service.py:33-44` | **Pin resolver classifies `uses:` by naive line-scanning, not the parser** — `stripped.startswith("uses:")` after `lstrip("- ")`. This both (a) misses `uses:` written in flow/JSON style or with unusual spacing, and (b) can mis-read a `uses:` substring inside a `run:` heredoc or a comment as a real action ref and fire a `get_commit_sha` against an attacker-influenced `owner/repo`. The actual rewrite (`operations._pin_steps`) uses the real AST, so the resolver and the rewriter disagree on what's a ref. | Resolve refs from the parsed AST (`wf.all_uses()`), not a text scan, so resolver and rewriter see the same set. |
| **Med** | security | `executor/service.py:79` + `github/client.py:112-156` | **No validation that branch/PR strings are safe before they hit the API.** `operation_id` flows into the branch ref `refs/heads/actionsplane/{operation_id}` and PR title/body. Today `operation_id` is server-derived (`{operation}-{id}`) and `operation` comes from `CampaignCreate`, so a campaign created via the (possibly unauthenticated) API can set `operation` to an arbitrary string — which becomes part of a git ref name and the PR title. A crafted value (newlines, `../`, control chars) could create an unexpected ref path or inject into the PR title/body. PR `body` interpolates repo-controlled `e.changes`/`rationale` unescaped (markdown injection, low impact). | Validate `operation` against the `OPERATIONS` registry keys at campaign-create time (operations.py:106 has the allowlist); sanitize/validate the branch component against `^[A-Za-z0-9._/-]+$`. |
| **Med** | security | `github/client.py:25-30` | **owner/repo still uninterpolated-but-unvalidated** (carried over). All write methods interpolate `owner`/`repo` straight into URLs. Campaign `repo_ids` are DB-sourced today, but with org-wide write scope this is the one spot worth a cheap `^[A-Za-z0-9._-]+$` guard to prevent a corrupted/crafted repo row from re-targeting the contents/refs/pulls API. | Validate `owner`/`repo` in `GitHubClient.__init__` call sites or a small helper. |
| **Med** | perf | `sync/worker.py:115-117,152-163` | **Audit/drift crons are serial and unbounded** (carried-forward High, re-noted because two more serial cron consumers — `audit_all`, `drift_sweep` — now share the pattern, multiplying the 5000/hr pressure: drift adds a `get_file_text` per matched workflow on top of audit's). No concurrency, no rate-limit backoff, no ETags. | Bound with `asyncio.Semaphore` + `gather`; add conditional requests + `Retry-After` handling in `GitHubClient`. |
| **Med** | perf | `executor/campaigns.py:64` | **Dry-run work is done twice.** `apply_campaign` re-runs the full `dry_run_repo` (list files + get each file + resolve every ref via network) for every target, duplicating all the I/O already done in `run_dry_run`. On a large campaign this doubles the GitHub call budget. | Persist the computed `FileEdit`s (or at least the resolved SHA map + blob SHAs) from dry-run and reuse on apply; re-fetch only the blob SHA to detect HEAD movement. |
| **Med** | perf/correctness | `api/app.py:255-263` | **SSE endpoint: no disconnect detection / subscriber cap** (carried forward). Each connection opens a dedicated Redis client + pubsub (bus.py:53). Many reconnecting dashboards = many connections; cleanup depends on the generator being cancelled. | Check `request.is_disconnected()`; cap concurrent subscribers; share one pubsub fan-out. |
| **Med** | security | `providers/gitlab/parser.py:81-92` | **GitLab include classification false-negatives.** (1) `_RESERVED` is missing real GitLab top-level keywords (`stages` present but no handling of nested includes; `include` entries using `inputs:` for components aren't validated). (2) A `project` include with `file:` but the analysis only reads `ref` — a project include pinned by branch vs tag isn't distinguished beyond the shared classifier. | **Low** | security | `ingestor/app.py:45` | Unbounded `request.body()` before checks — DoS (carried forward). | Enforce max content-length at proxy + in-app. |
| **Low** | correctness | `ingestor/app.py:60` | `json.loads(body)` unguarded -> 500 on malformed signed body (carried forward). | `try/except -> 400`. |

## 3. Top 5 to fix first

1. **Gate `apply` behind real authorization, not just a global boolean** (`executor/campaigns.py:46-49`, `api/app.py:228-241`). With `ACTIONSPLANE_API_TOKEN` unset by default, the org-wide PR path is reachable unauthenticated whenever `bulk_edits_enabled=true`. Make bulk edits require API auth to be configured (fail-closed) and record/require an approver. (High, security)
2. **Fix installation-token expiry in the campaign + worker caches** (`github/factory.py:31-37`, `executor/campaigns.py:23,51`, `worker.py:97-102`). Cache `InstallationToken` and re-mint on `is_expired()`; today long serial loops 401 mid-run. (High, perf/correctness)
3. **Make PR branches idempotent / re-appliable** (`executor/service.py:79`, `github/client.py:112`). Constant per-campaign branch name + `create_branch` 422-on-exists makes retries fail and can strand half-written branches. (High, correctness/safety)
4. **Make audit/drift/reconcile crons concurrent + rate-limit-aware** (`sync/worker.py`). Still the original High: serial, unbounded, no ETag/Retry-After. Now three cron consumers share it. (High, perf/scale)
5. **Pin from the dry-run-resolved SHAs, and resolve refs via the AST not a text scan** (`executor/service.py:28-45,64`). Closes the "approved diff ≠ committed diff" gap and the resolver/rewriter disagreement. (Med→High for a tool whose trust thesis is reviewable PRs)

## 4. Verified correct / solid

- **API auth wiring**: `require_token` is attached at the router level (`app.py:54`), so it applies to *every* `/api/v1` route including `/campaigns/*/apply` and `/events/stream` — no per-route omissions. Constant-time compare, correct `Bearer ` prefix handling, open-when-unset is intentional. The only gap is that "unset" is the default (see finding) and CORS is absent.
- **Webhook HMAC** (`ingestor/signature.py`): SHA-256 over raw body, `hmac.compare_digest`, rejects missing/bad-prefix headers; ingestor now **fails closed with 503** when no secret configured (`app.py:47-49`) and 401 on mismatch, before any processing. Correct.
- **App JWT / token handling** (`github/app_auth.py`): RS256, ~10-min TTL, `iat` backdated, `iss=app_id`; private key read from a path; tokens cached with a 60s expiry skew in `InstallationToken.is_expired`. No secrets/tokens/JWTs/payloads logged (grep-confirmed; the one audit log line logs only owner/repo/path).
- **Path-traversal hardening** (`github/client.py`): `..`-segment rejection + `quote(path)` on every contents/refs/commits call. The earlier latent SSRF/traversal vector is closed for `path`.
- **SQL safety**: entirely SQLAlchemy expression language + `pg_insert` ... `on_conflict_do_update`; no raw SQL, no f-string queries, parameterized throughout (repository.py). Upserts idempotent for at-least-once delivery.
- **YAML parsing**: `audit/parser.py` and `providers/gitlab/parser.py` both use `YAML(typ="safe")` — no arbitrary object construction. `executor/operations.py` uses round-trip mode only for *rewriting trusted-then-reviewed* content; safe.
- **DB/session lifecycle** (`db/base.py`): lazy cached async engine (asyncpg, no blocking driver), `async_sessionmaker(expire_on_commit=False)`, `get_session` proper `async with`. Worker uses a fresh `sessionmaker()` session per task invocation (`worker.py:40`, etc.) — correctly scoped, no shared-session races. (Note: there is no per-task-*within-gather* session because there is no gather.)
- **Audit lifecycle**: fingerprint dedup (`audit/findings.py`), reopen-on-reappear + resolve-when-absent (`repository.py:123-153`), per-file try/except so one bad workflow doesn't abort the repo audit (`audit/service.py:35-37`).
- **Indexes**: prior core indexes intact; new `ix_audit_findings_repo_resolved (repo_id, resolved_at)` (migration 0005) directly serves the hot `open_findings` path, and `ix_campaign_targets_campaign_id` serves `list_targets`. The `open_findings` `ORDER BY last_seen_at` is not covered by an index but is bounded by `limit` so acceptable.
- **`bulk_edits_enabled` default `False`** (`config.py:38`) — correct fail-safe default; the risk is purely when an operator flips it on without also setting an API token.
- **Concurrency refactor**: there is none — `sync/worker.py` is serial. So none of the hypothesized refactor bugs (shared mutable token-cache races under gather, session-per-task pool exhaustion, unbounded gather) exist. The token-cache *is* shared mutable state but is only ever touched by a single serial loop, so no race. If/when concurrency is added, the token cache and per-repo session scoping will need the usual care.
